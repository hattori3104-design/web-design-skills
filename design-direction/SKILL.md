# Design Direction Skill

記入済みヒアリングシートを入力に、デザイン方向性提案書（HTML下書き）を生成する。
思想は **半自動**：機械的に埋める部分と、デザイナーの判断が要る部分を明確に分け、後者は「案」または「要確認」として残す。

**必ず守る原則**
- 出力前に `branding-strategy` スキルの `references/branding-knowledge.md` を読み、WHY→HOW→WHATの論理と失敗例の知見を反映する。
- コンセプトは**断定せず2〜3案**出す。配色のhex・フォントは**候補**として出す。
- 出力末尾に**「人が確認すべき箇所」チェックリスト**を必ず付ける。

---

## Step 1: 入力シートを読む

ヒアリングシートの形式に応じて読み込む。

- **Googleスプレッドシート**：`Google Drive:read_file_content`（fileId指定）。
- **xlsx**：`extract-text file.xlsx` または pandas で全シートを読む。改善版は3シート構成（「ヒアリングシート」「トーン診断」「記入ガイド」）。**トーン診断シートの○の位置を必ず拾う**。
- 記入が薄い・空欄が多い場合は、その項目を「未記入（要ヒアリング）」として扱い、勝手に創作で埋めない。

読み取る主な対応（改善版シートの設問番号）：

| 提案書セクション | 参照する設問 |
|---|---|
| 0. ゴール | Q9–11 |
| 1. ターゲットと不安 | Q16–20 |
| 2. コンセプト（WHY） | **Q13–15**（どう思われたいか／価値・想い／人に例えると） |
| 3. トーン&マナー | **トーン診断シート** ＋ Q32 |
| 4. 配色 | Q33（＋Q13の印象） |
| 5. フォント | Q34（明朝/ゴシック寄り） |
| 6. 参考サイト | Q24–27 |
| 7. 変えない/変える | **Q36**（リニューアル時）＋ Q12 |
| 8. 次のステップ | Q37–38 |

---

## Step 2: 内部で方向性を組み立てる（出力前の思考）

`branding-knowledge.md` を踏まえ、頭の中で次を済ませてから書き始める。

1. **BtoB / BtoC 判定**（Q5）。BtoB＝知覚リスク極小化・QCDF・実績の可視化を軸に。BtoC＝自己一致性（Q20「こうありたい姿」）を軸に。
2. **コンセプト案を2〜3**。Q13–15を素材に、ゴールデンサークルのWHYを一言に昇華する。各案に「狙い」を1行添える。断定しない。
3. **トーンの言語化**。トーン診断シート（7軸）の○位置を、提案書テンプレの4軸トラックに集約して反映する。
   - **集約基準**：テンプレの4軸は診断の軸1〜4（堅実⇔革新／上品⇔親しみ／余白⇔情報量／無機質⇔手触り）をそのまま使う。残る軸5〜7（高級⇔身近／誠実⇔遊び心／力強い⇔やわらか）はトラック表示せず、コンセプト・配色・写真の判断材料として内部で使う。ただし案件で軸5〜7のどれかが極端（1または5）で方向を強く規定する場合は、近い4軸の1つと差し替えてよい。
   - **位置の%変換**：○位置 pos(1〜5) を `left` に変換する。目安は pos1→12% / 2→28% / 3→50% / 4→72% / 5→88%（端の見切れ防止のため0〜100%でなく12〜88%に寄せる）。
   - 4軸の○から、目指す形容詞3つ・目指さない形容詞を1行で言語化する。
4. **配色・フォントは候補**。「信頼を示すなら紺系（例 #1B3A5B 前後）」のように、方向＋例示hexを出す。確定でないと明示。
5. **失敗ガード**：リニューアル（Q3 / Q12 / Q36）なら、7章「変えない要素」を**必ず**埋める。Q36が空なら「要確認：消してはいけない目印の特定」と書く。表層的な全面刷新を無検証で推さない。

## Step 3: HTMLに流し込む

`assets/proposal-template.html` の**CSS・レイアウト・構造をそのまま使う**。各セクションの中身だけを入力シートの内容で差し替えて、新しいHTMLとして出力する。

### 3-0: config.json を読み、プレースホルダーを解決する（必須・最初に行う）

`config.json` を読み込み、以下の3つのプレースホルダーを実値に置換してから、案件内容の流し込みに進む。

| プレースホルダー | 参照するキー | 上書き条件 |
|---|---|---|
| `{{BRAND_NAME}}` | `brand.name` | ユーザーが「ブランド名は◯◯」と指定した場合のみ上書き |
| `{{DESIGNER_NAME}}` | `brand.designer` | ユーザーが「担当者名は◯◯」と指定した場合のみ上書き |
| `{{FOOTER_NOTE}}` | `brand.footer_note` | 通常は config の値をそのまま使う |

上書き指示の例（ユーザーがこう言ったら従う）：
- 「担当者名は山田で」→ `{{DESIGNER_NAME}}` を「山田」に
- 「ブランド名はデザイン工房ABCで」→ `{{BRAND_NAME}}` を「デザイン工房ABC」に
- 指示がなければ config.json の値をそのまま使う。**Sakumadesign をデフォルトにしない**。

差し替え時のルール：
- 冒頭の黄色い「記入例です」バナーは、**HTMLの`<div class="sample-banner">`本体と、`<style>`内の`.sample-banner`関連ルール（メディアクエリ内を含む）の両方**を削除する。HTMLだけ消すとCSSが残るので注意。
- 表紙のクライアント名・日付・担当を実値に。
- コンセプト（2章）は、テンプレの`.concept`ブロックを**案の数だけ複製**し、「A案 / B案」とラベルを付ける。
- 制作者メモ（`.note`）は、テンプレの「💡狙い」の趣旨を保ちつつ、その案件に即した内容に書き換える。クライアント提出時にトグルで隠せる仕様は維持する。
- 配色スウォッチのhex・フォントプレビューの書体は、Step 2で決めた候補を反映。フォントを変える場合は`<head>`のGoogle Fonts `<link>`とプレビューの`class`も合わせて変える。
- 未記入項目は、テンプレの`.fill`（斜線の記入枠）スタイルを使って「要ヒアリング」と分かる形で残す。創作で埋めない。

出力は `/mnt/user-data/outputs/` にHTMLとして保存し、`present_files` で渡す。

## Step 4: 「人が確認すべき箇所」チェックリストを添える

HTML本体とは別に、チャット応答の中で必ず以下の形のチェックリストを出す。自動生成の限界を明示し、デザイナーの判断を促す。

```
## 確認・調整が必要な箇所
- [ ] コンセプト：A/B案のどちらを採るか（または統合）
- [ ] 配色：候補hexは方向性のみ。実際の画面で最終調整を
- [ ] フォント：候補から確定を
- [ ] 参考サイト：URLの採否はデザイナー判断で
- [ ] （リニューアル）変えない要素：Q36の内容で過不足ないか
- [ ] 未記入だった項目：〔具体名〕の追加ヒアリング
```

該当のない項目は省き、未記入だった実際の項目名を列挙する。

## 品質チェック（出す前に自問）

- [ ] `config.json` を読み、プレースホルダー3つ（BRAND_NAME / DESIGNER_NAME / FOOTER_NOTE）をすべて解決したか。`{{` が残っていないか
- [ ] `branding-knowledge.md` を読み、WHY→HOW→WHATの順序で構成したか
- [ ] BtoB/BtoCの軸を取り違えていないか（BtoB=知覚リスク／BtoC=自己一致性）
- [ ] コンセプトを断定せず複数案にしたか／配色・フォントを「候補」と明示したか
- [ ] リニューアルで「変えない要素」を埋めた（または要確認とした）か
- [ ] 空欄を創作で埋めず「要ヒアリング」として残したか
- [ ] 末尾に確認チェックリストを付けたか
