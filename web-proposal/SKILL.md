# ウェブ制作 提案書作成スキル

## このスキルの役割

web-strategy（戦略・サイトマップ）やヒアリングシートで固まった中身を、**クライアントの合意を取るための提案書**に組み立てる。考える作業ではなく、組み立てて文書化する作業。

| インプット | web-strategyの成果物／ヒアリングシート／会話中の戦略情報 |
|------------|--------------------------------------------------------|
| アウトプット | ①構成案（人間が確認）→ ②本文 → ③HTMLプレビュー（人間が確認）→ ④ファイル（.pptx ＋ .md／読む文書が要るときは .docx） |

応答ルール：結論を先に出す。事実と推測を分ける。経営判断にはリスクとダウンサイドを併記する。法務・税務・労務にあたる論点は専門家への確認を促す。専門用語の補足は通じにくい隠語・略語だけにとどめ、くどくしない。

---

## ステップ0：準備と論点詰め（最重要・必ず先に行う）

**最初に `config.json` を読む（必須）。** `author.studio`・`author.name`・`author.footer_note` の3値を取得し、HTMLプレビュー・.docx・.pptx のすべての「作成者」表示に使う。案件ごとに担当者を変えたい場合は「担当者は○○」と指示すれば `author.name` だけ上書きする。`{{` が出力に残った場合は config.json の読み込みが失敗しているので再確認する。

**次に web-strategy の成果物を探して読み込む（必須）。** 提案書の中身は web-strategy が作った戦略を素材にする。以下の順で素材を集める：

1. `/mnt/user-data/outputs/` に `web-plan-*.md`（web-strategy の出力ファイル）があれば**必ず読み込み**、プロジェクト背景・ターゲット・ファネル・ポジショニング・サイトマップ・競合調査を提案書の素材として使う。
2. ファイルが無ければ、**同じ会話の中に web-strategy の成果（戦略・サイトマップ等）があるか**を確認し、あればそれを使う。
3. どちらも無ければ、戦略がまだ無い状態。**提案書を書き始める前に**「先に web-strategy で戦略・サイトマップを固めますか？それとも今ある情報だけで進めますか？」と確認する（戦略が無いまま提案書だけ作ると中身が薄くなる）。

> 読み込んだ web-strategy の内容は、提案書用に**要約・再構成**して使う。全文をそのまま貼らない（戦略は要点の再確認にとどめ、後半の設計・進行に重心を置く）。

このスキルは**受注確定後に使う前提**。タイプ選択はしない。1本の提案書に、**受注前提案の良さ（方向性・ビジョンで「この方向でいきましょう」と惹きつける）**と、**キックオフ設計提案（設計・進行・スコープで「この内容で進めます」と握る）**の両方を併せ持たせる。

| 前半（口説く・方向性） | このプロジェクトで目指す姿・狙い・なぜこの設計か。受注前提案の熱量を引き継ぐ |
|---|---|
| 後半（握る・設計と進行） | サイトマップ・要件・スコープ・スケジュール・体制。キックオフ設計提案として合意を取る |

`references/proposal-patterns.md` を読み、**構成テンプレ**と**詰めるべき論点チェックリスト**に沿って、不明点を1つずつ確認する。特に次の3点は提案書の成否を分けるので必ず詰める：

1. **この提案書で引き出したい承認は何か**（着地点が決まる）
2. **見積もりは載せない（既定）**。代わりにスコープ表＋「含まないもの」＋分担表で「ここまでが今回分」という歯止めを作る。クライアントから明示的に金額提示を求められたときのみ載せる
3. **納期・期限の前提**（締切があるなら、クライアント側の素材提出・確認の締切を「前提条件」として握る）

> 論点を詰める作業は grill-me スキルの進め方（1問ずつ・推奨回答を添える）と相性がよい。ユーザーが「詰めて」と言ったら、そのスタイルで論点を潰す。

## ステップ1：構成案の出力（人間が確認）

論点が固まったら、**ファイルはまだ作らず**、構成案（目次）をチャットに出して承認を取る。

### 出力フォーマット

```
## 提案書 構成案

【形式】.pptx（＋ .md 併産）／読む文書が要るときは .docx
【引き出す承認】[このページで合意したいこと]

【構成】
1. [セクション名] … [このセクションで伝えること]
2. ...
（以降、proposal-patterns.md のテンプレをベースに案件に合わせて増減）

【今回の重要ポイント】
- 見積もり：載せない → スコープ表＋「含まないもの」＋分担表で歯止め
- 期限：[前提条件として握る内容]
- リスク併記：[クライアントに共有すべき前提・リスク]

この構成でよければ「OK」とお伝えください。本文を書きます。
```

各タイプのセクション構成は `references/proposal-patterns.md` を参照。全部入れず、案件に必要なものだけ選ぶ。

---

## ステップ2：本文の生成

構成案が承認されたら本文を書く。書き方の原則：

- **結論先出し**：各セクションの冒頭に「何を言うか」を置く
- **クライアントの言葉を使う**：ヒアリングにあった表現を活かす
- **リスクを隠さない**：目標が背伸びなら「これだけでは届かない」と前提を共有する。信頼につながる
- **読み手はクライアントの意思決定者**：判断しやすい言葉で。表と箇条書きを活用
- **数値は出典の主張として扱う**：業界の一般値は「〜という傾向がある」と添え、自社実測値と区別する

本文はチャットに出すか、長ければ .md ファイルに書き出して確認を取る。**.md は最終アウトプットとしても常に併産する**（編集・再利用しやすい）。

---

## ステップ2.5：HTMLプレビュー（人間が確認・必須）

本文が固まったら、**.docx / .pptx / .md を作る前に必ず HTML でビジュアルを見せる**。クライアントに渡す前に「構成・文言・見た目の方向性」をデザイナー自身が目視確認するための工程。

### HTMLの仕様

| 項目 | 内容 |
|------|------|
| スタイル | **スライド風**（横長・1セクション＝1枚カード）を基本（既定の .pptx に合わせる）。.docx で出すときのみ**文書風**（縦長・A4イメージ） |
| 色 | ネイビー基調（見出し `#1F3A5F`、背景 `#F2F4F7`）。クライアントのブランドカラーがあれば反映 |
| フォント | 游ゴシック → Noto Sans JP フォールバック |
| 内容 | 本文のすべてのセクションを入れる。ダミーテキストは使わない |
| ファイル名 | `[クライアント名]_提案書_preview.html` |

### 確認を取る文言

```
HTMLプレビューを出力しました。ブラウザで開いて以下を確認してください。

確認ポイント：
- [ ] セクションの順番・抜けがないか
- [ ] 文言の方向性・トーンに違和感がないか
- [ ] 金額・日付・クライアント名に誤りがないか

問題なければ「OK」で .docx / .pptx の生成に進みます。
修正があれば箇所を教えてください。
```

承認前にファイル生成に進まない。

---

## ステップ3：ファイル生成

既定は **.pptx**（プレゼンしながら合意を取る）。**.md は常に併産**。読んで合意する文書が要る場合のみ **.docx** も出す。

### 既定：方向性＋設計を1本にした提案書 → .pptx

**pptx スキル（`/mnt/skills/public/pptx/SKILL.md`）と `references/pptx-guide.md` を読んでから作る。** プレゼン資料なので、文書とは別の作法（1スライド1メッセージ・視覚要素・大きな数字）が要る。pptxgenjs で生成し、視覚QAまで行う。**前半は方向性・狙いで惹きつけ、後半はスコープ・スケジュール・体制で握る**構成にする。

### 任意：読んで合意する文書が要るとき → .docx

同梱の `scripts/build_docx.js` をテンプレートとして使う。**docx スキル（`/mnt/skills/public/docx/SKILL.md`）の作法に従う。**

```bash
cd <作業ディレクトリ>
npm install docx          # 未インストールなら
node build_docx.js        # CONTENT配列を案件内容に書き換えてから実行
python /mnt/skills/public/docx/scripts/office/validate.py proposal.docx
```

`build_docx.js` に作り込んだ要点（崩れ対策・そのまま踏襲すること）：

- **表紙は独立セクション**にしてヘッダー／フッターを出さない
- **テーブル行は `cantSplit: true`** にして改ページでの分割崩れを防ぐ。行数の多い表は章の前に改ページを入れて1ページに収める
- **日本語フォントは游ゴシック（Yu Gothic）** を既定にする。英数字混在でも崩れにくい
- **テーブル幅は必ず DXA 指定**（PERCENTAGE は Google Docs で崩れる）。`columnWidths` の合計＝テーブル幅＝本文幅（US Letter・1インチ余白で 9360）
- **コールアウトボックス**（左罫線アクセント＋淡い背景）で「前提条件」「リスク共有」を強調する
- カラーはネイビー基調（見出し `1F3A5F`、ヘッダ背景 `DCE6F1`、ボックス背景 `F2F4F7`）。クライアントのブランドカラーがあれば差し替える

生成後は必ず PDF 化して目視確認する（表紙・表の分割・コールアウトの3点を重点チェック）：

```bash
python /mnt/skills/public/docx/scripts/office/soffice.py --headless --convert-to pdf proposal.docx
pdftoppm -jpeg -r 100 proposal.pdf page
```

### 共通

- 完成したら `/mnt/user-data/outputs/` にコピーし、`present_files` で共有する
- ファイル名は `[クライアント名]_提案書.docx` など分かりやすく

## やりがちな失敗（先に防ぐ）

| 失敗 | 対策 |
|------|------|
| いきなり本文から書き始める | まず構成案（目次）で承認を取る。前半=方向性／後半=設計の2部構成を外さない |
| 見積もりをデフォルトで入れる | この提案書は金額を載せないのが原則。スコープ表＋「含まないもの」＋分担表で「ここまでが今回分」の歯止めを作る。金額提示はクライアントから明示的に求められたときのみ |
| 金額を外したのにスコープも書かない | 金額表は「ここまでが今回分」の歯止めでもある。外すならスコープ表＋「含まないもの」＋分担表で代替する |
| 締切だけ約束して前提を握らない | 制作遅延の多くは素材待ち・確認待ち。公開日は「素材提出◯日・確認◯営業日以内」を前提条件として明記する |
| 受注済みの相手に戦略をフル再説明 | キックオフでは戦略は要点の再確認にとどめ、要件・スケジュール・体制に重心を置く |
| docxの表が改ページで崩れる | `cantSplit` と章前改ページで対処（build_docx.js に実装済み） |
