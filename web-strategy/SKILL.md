# Web Planning Skill

Webサイトのゼロからのプロジェクトプランニングとサイトマップ提案を行う。
調査 → 戦略立案 → コンテンツ設計 → サイトマップという流れで進める。

---

## Step 0: ヒアリング（必須）

スキル起動時、まず以下を確認する。未回答の項目があれば `ask_user_input` で質問する。

```
必須ヒアリング項目：
1. クライアント・プロジェクト名
2. 業種・提供サービス/製品
3. BtoB / BtoC / 両方
4. サイト制作の目的（新規 / リニューアル / LP追加 など）
5. ターゲットユーザー像（簡単でOK）
6. 競合・参考にしたいサイト（あれば）
7. アウトプット形式（Markdown / Word(.docx) / チャット内テキスト）
8. 予算感・運用体制（小規模／中規模／大規模の判定用。専任担当の有無など）
```

**BtoB（または両方）の場合は、戦略立案に入る前に `references/btob-web-strategy.md` を読み込むこと。**
そこに記載された「回答前の内部思考」「案件規模の判定」「戦略・実行・失敗例」の知見を、以降のすべてのアウトプットに具体的に織り込む。一般論で埋めず、知識ベースの具体策・数値・フレームを使う。

**成果改善・CVR・表示速度・計測・EC運用が論点のとき（特に既存サイトの診断/改善）は、`references/conversion-optimization.md` を読み込むこと。** 新規制作でも、Step 1のファネル設計やワイヤー要点を詰める前に読むと、行動科学・速度・計測の土台を踏まえた設計ができる。EC節（カゴ落ち・客単価・UGC）はECサイトのときだけ適用し、BtoBコーポレートサイトには持ち込まない。

ヒアリング完了後、以下の順にアウトプットを作成する。

## Step 1: プロジェクトプラン

### 1-1. プロジェクト背景

以下の構成でまとめる：

```markdown
## プロジェクト背景

### 現状と課題
- 現在のWebサイト（または状況）の課題
- ビジネス上の背景・変化

### プロジェクト目的
- このWebサイトで達成したいこと
- 成功指標（KPIイメージ）

### ターゲットユーザー
- プライマリ：〇〇（属性・ニーズ）
- セカンダリ：〇〇（属性・ニーズ）
```

### 1-2. チャネル概要

ユーザーがどのチャネルからサイトに流入するかをまとめる：

```markdown
## チャネル概要

| チャネル | 役割 | 優先度 |
|---------|------|--------|
| 自然検索（SEO） | ... | 高/中/低 |
| 広告（リスティング/SNS） | ... | 高/中/低 |
| SNS・コンテンツ | ... | 高/中/低 |
| 口コミ・紹介 | ... | 高/中/低 |
| メール・CRM | ... | 高/中/低 |

### 重点チャネル戦略
- 〇〇チャネルを重視する理由と施策概要
```

### 1-3. ファネル概要

認知→興味→検討→行動のファネルを設計する：

```markdown
## ファネル概要

### ファネルマップ

認知（AWARE）
  └→ 流入元：〇〇
  └→ コンテンツ：〇〇ページ

興味・関心（INTEREST）
  └→ 提供価値：〇〇
  └→ コンテンツ：〇〇ページ

検討（CONSIDER）
  └→ 差別化ポイント：〇〇
  └→ コンテンツ：事例・料金・比較など

行動（ACTION）
  └→ CV地点：〇〇（問い合わせ / 購入 / 資料DL など）
  └→ CTA設計：〇〇

### CVR向上のための設計ポイント
- 〇〇
```

### 1-4. ポジショニングマップ概要

**競合調査データ（Step 2）と組み合わせて作成する。**

```markdown
## ポジショニングマップ概要

### 軸の設定
- X軸：〇〇（例：価格 高⇔低、ターゲット 個人⇔法人）
- Y軸：〇〇（例：機能 シンプル⇔多機能、対応 汎用⇔専門）

### 競合ポジション
| プレイヤー | X軸 | Y軸 | 特徴 |
|-----------|-----|-----|------|
| 自社（目指す位置） | ... | ... | ... |
| 競合A | ... | ... | ... |
| 競合B | ... | ... | ... |

### 自社の差別化ポイント
- 〇〇の領域で独自ポジションを確立できる
```

## Step 2: 同業他社コンテンツ調査

### 調査方法の選択

| 状況 | 方法 |
|------|------|
| 競合URLをユーザーが提供 | `web_fetch` でページ内容を取得・分析 |
| 競合URLが不明 | `web_search` で「[業種] [キーワード] サイト」などで検索して上位サイトを特定し `web_fetch` |
| 両方 | 提供URLを優先し、不足分は検索で補完 |

### 調査観点

各競合サイトについて以下を調査する：

```markdown
## 競合サイト調査：[サイト名]
URL: 〇〇

### サイト構造
- 主要ページ構成（ナビゲーション等から把握）
- メインCTA

### コンテンツの特徴
- 強調しているメッセージ・価値提案
- コンテンツカテゴリ（事例・ブログ・料金 など）
- 差別化ポイントとして訴求していること

### UX・デザイン
- 第一印象・雰囲気
- 特徴的なUI要素

### 評価まとめ
- 参考にすべき点
- 自社が勝てる点・差別化余地
```

```markdown
| ブログ/メディア | | | | |
| 差別化訴求 | | | | |
```

## Step 3: コンテンツ案

競合調査とファネル設計を踏まえて、必要なページ・コンテンツを一覧化する：

```markdown
## コンテンツ案

### 必須コンテンツ（Must Have）
| コンテンツ | 目的 | ファネル段階 | 備考 |
|-----------|------|------------|------|
| トップページ | 全体訴求・導線 | 認知〜行動 | |
| サービス/製品ページ | 詳細説明 | 検討 | |
| 会社概要 | 信頼醸成 | 検討 | |
| お問い合わせ | CV | 行動 | |

### 推奨コンテンツ（Should Have）
| コンテンツ | 目的 | ファネル段階 | 備考 |
|-----------|------|------------|------|
| 導入事例/実績 | 信頼・具体化 | 検討 | |
| ブログ/コラム | SEO・認知 | 認知〜興味 | |
| FAQ | 不安解消 | 検討 | |
| 料金ページ | 検討促進 | 検討〜行動 | |

### オプションコンテンツ（Nice to Have）
| コンテンツ | 目的 | ファネル段階 | 備考 |
|-----------|------|------------|------|
| 資料ダウンロード | リード獲得 | 検討 | BtoBで有効 |
| ホワイトペーパー | 専門性訴求 | 興味〜検討 | |
| 動画コンテンツ | 理解促進 | 興味 | |
```

## Step 4: サイトマップ案

コンテンツ案を階層構造に整理する：

```markdown
## サイトマップ案

### 構造図

HOME（トップページ）
│
├── サービス / 製品
│   ├── サービスA
│   ├── サービスB
│   └── 料金・プラン
│
├── 導入事例 / 実績
│   ├── 事例一覧
│   └── 事例詳細（× N件）
│
├── 会社情報
│   ├── 会社概要
│   ├── ミッション・ビジョン
│   └── 採用情報（オプション）
│
├── ブログ / コラム（オプション）
│   ├── 記事一覧
│   └── 記事詳細
│
├── よくある質問（FAQ）
├── お問い合わせ
└── プライバシーポリシー / 利用規約

### ページ優先度・制作順序

| フェーズ | ページ | 優先度 |
|---------|--------|--------|
| Phase 1（MVP） | HOME、サービス、会社概要、お問い合わせ | 最高 |
| Phase 2 | 事例、料金、FAQ | 高 |
| Phase 3 | ブログ、資料DL、その他 | 中 |

### URL設計（案）

| ページ | URL |
|--------|-----|
| トップ | / |
| サービス | /services/ |
| 事例一覧 | /cases/ |
| 会社概要 | /company/ |
| お問い合わせ | /contact/ |
```

## 注意事項

- **web_search / web_fetch を積極的に使う**：競合調査は実際にサイトを取得して分析すること。推測で埋めない。
- **業種に合わせてカスタマイズ**：BtoB向けはリード獲得・事例・料金訴求を重視。BtoC/ECは購買導線・商品ページを重視。
- **ポジショニングマップの軸**：業種・競合状況によって適切な軸を選定する（価格、ターゲット層、専門性、対応範囲など）。
- **コピーライティングは references/copywriting.md を必ず使う**：VoCリサーチ・ベネフィット3段階・PASONA/QUEST・マイクロコピー・ABテスト設計の知見を具体的に織り込む。「デザインを先に磨く」「抽象的ポエム訴求」などの失敗例に当たる依頼はリスクを先に伝える。
- **段階的に提案する**：全部一度に出さず、「プロジェクトプラン」→「競合調査」→「サイトマップ」の順に確認を取りながら進めると良い。
- **BtoBは references/btob-web-strategy.md を必ず使う**：KPI逆算・3層ペルソナ・ファネル別コンテンツ設計を、案件規模に応じた深さで適用する。小規模案件にMA/CRM連携やヘビーコンテンツを過剰提案しない。
- **失敗例に該当する依頼はリスクを併記**：ユーザーの依頼が知識ベースの「失敗例」（要件定義のスキップ、ちゃぶ台返し、SEO断絶など）に当たる場合、望まれていてもまずリスクと対策・代替案を示す。
- **数値は出典の主張として扱う**：知識ベースの定量値は元資料の主張であり検証済みの事実ではない。クライアント提案では断定を避け、自社案件の実測値と区別する。
- **専門用語は噛み砕く（ただしくどくしない）**：読み手はデザイナー本人とクライアント（経営者・リーダー）。説明を添えるのは「EFO」「301リダイレクト」など通じにくい隠語・略語だけにし、「見込み客」「事例」「問い合わせ」など通じる言葉は素のまま使う。括弧の重ね使いや毎回の前置きは避け、簡潔に。
