# ワイヤーフレーム作成スキル

## このスキルの役割

web-strategy で固まった「誰に・何を・どの順番で見せるか」を、ページの骨組み（ワイヤーフレーム）として可視化する。デザインの前段階として、**構造・コンテンツ・CTA導線**を確定させることが目的。

| インプット | web-strategyの成果物（サイトマップ・ファネル・3層ペルソナ・強みの要点） |
|------------|-----------------------------------------------------------------------|
| モードA出力 | HTMLワイヤーフレーム（グレースケール・アノテーション付き） |
| モードB出力 | Figma Make用テキストプロンプト（.md） |

---

## ステップ0：モード選択と前提確認

起動時、まず以下を確認する。会話の流れで分かる場合は聞かない。

```
確認事項：
1. 対象ページ（トップ / サービス / 会社概要 など。複数ある場合は1ページずつ進める）
2. モード（A:HTMLワイヤー / B:Figma Makeプロンプト / A→B両方）
3. トーン＆マナーの指定（あれば。例：「信頼感・先進的・クリーン」）
   ※ モードBでは必須。モードAはグレースケールなので任意
4. web-strategyの成果物が会話にあるか（なければ要点を聞く）
```

モードが「A→B両方」の場合、Aを作って確認を取ってからBに進む。

## モードA：HTMLワイヤーフレームの生成

詳細は `references/html-wireframe-guide.md` を読んでから作業に入る。
以下はモードAの要点。

### 基本方針

- **グレースケール限定**：色は使わない（`#F7F7F7`〜`#222222` の範囲）。デザインの議論を構造の議論と切り離すため
- **アノテーション付き**：各セクションに黄色の注釈ボックス（`.wf-note`）を配置し、ファネルの位置づけ・確認事項・素材の出所を記載
- **戦略設計を忠実に反映**：ファネル（認知→興味→検討→行動）の各段階に対応するセクションを配置。CTAはファネル「行動」に対応するセクション全てに入れる
- **クライアント打ち合わせ資料として使える品質**：アノテーションで「ここはクライアントへの確認事項」を明示する

### セクション構成の組み立て方

web-strategyのファネル設計とサイトマップをベースに、以下の優先順位でセクションを選ぶ：

| 優先度 | セクション | ファネル | 入れる条件 |
|--------|-----------|---------|-----------|
| 必須 | HEADER（グローバルナビ） | — | 常に |
| 必須 | HERO | 認知〜興味 | 常に |
| 必須 | 強み・選ばれる理由 | 興味 | 常に |
| 必須 | メインCTA帯 | 行動 | 常に |
| 必須 | FOOTER | — | 常に |
| 推奨 | 実績数字（stat strip） | 興味〜検討 | 数字で信頼を示せるとき |
| 推奨 | 主要サービス・事業内容 | 検討 | 製造・サービス・士業など |
| 推奨 | 品質・認証・実績 | 検討 | 信頼証明が必要なとき |
| 推奨 | 取引実績・事例 | 検討 | 社名・実績が出せるとき |
| 推奨 | FAQスニペット＋サイドCTA | 検討〜行動 | 問い合わせ前の不安が多いとき |
| 任意 | ニュース・お知らせ | 認知 | 更新頻度が高い場合のみ |

**CTA設計の鉄則**：
- 全ページ共通：HEADER右端にCTAボタンを配置（sticky固定）
- トップページ：HERO・FAQサイドカード・メインCTA帯の最低3箇所
- その他ページ：各ページの末尾と、コンテンツ途中の自然な区切りに1〜2箇所

### 生成・出力

```bash
# /mnt/user-data/outputs/ に保存して present_files で共有
# ファイル名: wireframe-[ページ名].html
# 例: wireframe-index.html / wireframe-services.html
```

生成後、アノテーションの「クライアントへの確認事項」をチャットで箇条書きにまとめて伝える。打ち合わせのチェックリストとして使えるため。

## モードB：Figma Makeプロンプトの生成

詳細は `references/figma-make-guide.md` を読んでから作業に入る。
以下はモードBの要点。

### 基本方針

- **英語で書く**：Figma Makeは英語プロンプトの方が精度が高い。テキストコンテンツ（日本語コピー）は日本語のまま渡してよい
- **セクションごとに分割して記述**：`## SECTION N — セクション名` の形式で構造化する
- **色・フォント・サイズを具体的に指定**：HEXカラーコード・px値・font-weightを明示する。曖昧な指示（"dark blue"など）は生成結果がブレる
- **プレースホルダーの指示を入れる**：写真・ロゴが入る場所は「Gray rectangle (W×H px), label: '○○の写真（支給）'」と明示
- **NGパターンを明示する**：装飾的なカラーバー・タイトル下アクセントライン・AI生成っぽいパターンを避けるよう末尾の補足指示に入れる

### トーン＆マナーの翻訳

クライアントの言葉（日本語のトーン指定）を、Figma Makeに渡せる英語の設計言語に変換する。
翻訳パターンは `references/figma-make-guide.md` の「トーン変換表」を参照。

### プロンプトの構成

```
## 全体方針
- ブランド・事業の説明（1段落）
- Visual direction（トーン・カラーパレット・タイポグラフィ・レイアウト・Signature element・NGパターン）

## SECTION 1 — HEADER
...

## SECTION N — FOOTER

## 補足指示（Figma Make向け）
- フレームサイズ・コンテンツ幅
- Auto-layout指定
- スペーシングのグリッド（8px基準など）
- CTAボタンの形状（border-radius統一など）
```

### 出力

```bash
# /mnt/user-data/outputs/ に保存して present_files で共有
# ファイル名: figma-make-prompt-[ページ名].md
# 例: figma-make-prompt-index.md
```

## やりがちな失敗（先に防ぐ）

| 失敗 | 対策 |
|------|------|
| ワイヤーに色をつけてしまう | モードAは`#F7F7F7`〜`#222222`のグレースケール限定。色の判断はデザイン工程で行う |
| ファネルを無視してセクションを並べる | 必ず「このセクションはファネルのどの段階か」を意識してアノテーションに記載する |
| CTAが少ない | サイトのCTAは最低3箇所（HEADER・コンテンツ中・ページ末尾）。「図面添付できる」（製造業）「30分の無料相談」（士業・コンサル）など、業種に合わせた具体的なCTAコピーが効く |
| Figma Makeプロンプトが曖昧 | 「navy」ではなく「#0B2545」。「large」ではなく「46px bold」。数値で指定する |
| 1ページに詰め込みすぎる | セクションは7〜9個が目安。それ以上になるなら別ページに分離できないか検討する |
| 写真・素材の扱いが曖昧 | アノテーションに「支給素材」「クライアント確認が必要」を必ず明記する |
